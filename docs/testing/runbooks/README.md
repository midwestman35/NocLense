**Sub-phase:** 07J.2 (Batch D, Phase 07 Tauri migration)
**Parent spec:** `docs/superpowers/specs/2026-04-23-phase-07j-review-and-test-infra-design.md`
**Inventory:** `docs/testing/standalone-inventory.md`
**Started:** 2026-04-23

## Purpose

Manual test protocols for every screen in NocLense standalone. Each runbook is a pass/fail checklist a human (or the `noclense-qa-runner` agent built in 07J.3) can walk to reproducibly verify behavior.

Runbooks are the durable QA output of Phase 07. They survive the merge, seed the 07J.3 automation harness, and become the reference for every future change that touches a covered surface.

## How to use

- **Human runs:** open a runbook, launch `npm run tauri:dev` (or install the built bundle), walk each step, tick pass/fail. A fail at any step means halt — the finding goes into the inventory or a new issue.
- **Automated runs (post-07J.3):** the `/smoke-tauri <runbook-name>` slash command drives the Playwright-Tauri harness through the same steps, captures screenshots, and writes a report to `tools/claude-automation/reports/`.

## Conventions

Each runbook follows the same skeleton, spec'd in the 07J slice plan §2.07J.2:

1. **Preconditions** — fixture data, required app state, credentials, OS setup. Explicit; no "clean install" shorthand.
2. **Steps** — numbered actions with one user input per step. Every step pairs with a specific observable outcome (not "screen loads" — something like "header renders text 'NocLense' and button with accessible name 'Continue'").
3. **Pass criteria** — explicit per-step. If a step has multiple assertions (e.g. both layout and behavior), each gets its own bullet.
4. **Known failure modes** — a running list of regressions that have broken the surface before, seeded from Phase 06 and Phase 07 incidents. New failures land here after resolution so the next reviewer recognizes the smell.
5. **Automation target** — which steps 07J.3's Playwright driver can execute, which stay manual, and why (subjective fidelity, external service dependency, OS permission gate, etc.).

### Pass-criterion precision

A pass criterion answers: *"If this assertion passed, what visible evidence proves it?"*

| ❌ Too vague | ✅ Specific enough |
|---|---|
| Dashboard renders | Card with heading `"Recent Investigations"` appears; at least 1 row visible if `caseLibraryService.getAll()` returns ≥1 case |
| User can sign in | Splash's Continue button fires `onContinue`; DashboardScreen mounts within 300ms; no console errors |
| Log stream works | `LogViewer` renders with `role="grid"`; virtualized rows populate as user scrolls from 0 to 100,000-th entry with no frame drop >50ms |

### Fixture discipline

Log fixtures live under `tools/claude-automation/fixtures/` (created in 07J.3). Sample files:

- `small-sip.log` — ~5MB, 5k entries, covers standard SIP methods.
- `medium-mixed.log` — ~25MB, 25k entries, mixed SIP + Datadog-style JSON.
- `large-streamed.log` — >100MB, 100k+ entries, generated by a script — NOT checked into repo (too big). Generator script lives at `tools/claude-automation/fixtures/generate-large.ts`.

For manual runs before 07J.3 lands, Enrique's real case exports are the fixture set.

## Staying current

Every runbook is versioned alongside the code it covers. When a UI change lands that affects a runbook:

1. Update the runbook step(s).
2. Add a row to the runbook's "Known failure modes" section if the change was a regression/fix.
3. Note the slice commit SHA in a revision row at the top.

A runbook that doesn't match current behavior is worse than no runbook — it teaches the wrong shape.
